# AI Q版泡泡堂 · 项目总结

> 一个浏览器端多人对战游戏，致敬经典泡泡堂玩法。  
> 开发周期：1天 | 技术栈：Node.js + WebSocket + Phaser.js

---

## 一、项目概述

**目标**：做一个能在浏览器直接打开的Q版泡泡堂对战游戏，支持真人联机、Bot对战、Agent接入。

**核心特色**：
- 真人联机：创建房间 → 分享链接 → 好友直接加入
- 内置Bot：4种难度（简单/中等/困难/疯子）
- 战斗日志：实时记录炸弹放置、击杀、道具拾取
- 多模式：1v1、FFA混战、2v2组队、人机挑战

## 三、核心功能

### 3.1 游戏机制
- 13×11地图：石头（不可破坏）、砖块（可破坏，概率掉道具）、空地
- 炸弹：3秒倒计时，十字形爆炸，遇石头阻挡
- 道具：炸弹+1、火力+1、速度+1、踢炸弹、穿墙（限时）
- 命数：初始2命，被炸-1命，归零出局，开局3秒无敌

### 3.2 联机系统
- 创建房间自动生成4位房间号
- 浏览器地址栏自动变成 `?room=XXXX`，直接分享链接即可邀请
- 通过分享链接进入网页自动加入房间
- 首页显示当前所有可加入房间（以房主名字命名）

### 3.3 Bot AI
| 难度 | 特点 |
|------|------|
| 简单 | 随机走动，偶尔放炸弹 |
| 中等 | 躲避炸弹，捡道具 |
| 困难 | 追击敌人，计算安全区域 |
| 疯子 | 贴脸放炸弹，激进追击 |

### 3.4 战斗日志
实时推送事件：炸弹放置、玩家被炸、击杀、道具拾取、游戏开始/结束。

## 六、开发心得

### 6.1 关于"服务端权威"

泡泡堂这类游戏必须让后端持有游戏状态权威。所有炸弹计时、爆炸计算、碰撞检测都在服务端执行，客户端只负责渲染。这保证了：
- 防止作弊
- Agent和玩家看到一致状态
- 实现简单，不易出现不同步

### 6.2 关于程序化纹理

MVP阶段不需要等待美术资源。用 Phaser 的 `Graphics` 动态生成地面、石头、砖块、炸弹、火焰纹理，代码即美术。整个前端 < 100KB，零外部图片依赖。

### 6.3 关于邀请链接

最初设计是复制4位房间号手动输入，体验很差。改成浏览器地址栏自动更新为 `?room=XXXX` 后，用户直接复制浏览器地址发给朋友即可，零门槛。这是从"功能可用"到"体验流畅"的关键改进。

### 6.4 关于Bot AI

不需要复杂算法。4种难度的核心区别只有3个参数：
- 是否躲避炸弹
- 是否追击敌人
- 放炸弹的概率

简单规则 + 不同参数 = 明显的难度差异。AI设计不需要过度工程化。

### 6.5 关于端口问题

开发过程中反复遇到端口被占的问题。根本原因是：用 `&` 或 `timeout` 在后台启动的测试进程没有正确关闭。解决方案：
- 固定使用一个确认空闲的端口
- 启动前检查占用情况
- 开发测试脚本用完后立即 kill

### 6.6 关于状态同步

不需要客户端预测+回滚（FPS那种），泡泡堂是慢节奏策略游戏。服务端每100ms广播一次完整状态，延迟完全可接受，实现却简单得多。

## 八、Bug修复与优化记录（第2轮迭代）

### 8.1 阻断级 Bug

| 现象 | 根因 | 修复 |
|------|------|------|
| 按钮全死、卡在"连接中" | game.js 绑定了 HTML 中不存在的 `btn-join-tab`，顶层抛错导致整个脚本（含 WS 连接）中止 | 补按钮 + 所有绑定走 `$on()` 防御式函数，单个元素缺失只警告不崩溃 |
| 游戏画面全黑 | `sprite.add()` 不存在（只有 Container 有），异常打断场景 `create()`，游戏循环卡死 | 玩家名字文本改挂 entityContainer，逐帧跟随 |
| 刷新分享链接 404 | 服务器把 `?room=XXXX` 查询串当文件名读 | 只取 URL 路径部分 |
| Phaser 偶发加载失败致全站瘫痪 | 依赖 jsdelivr CDN | phaser.min.js 下载到本地 `public/js/lib/`，零外部依赖 |

### 8.2 联机逻辑 Bug

| 现象 | 根因 | 修复 |
|------|------|------|
| 同一身份在房间里出现两次 | 重复建房/入房不退出旧房间；同名可重复加入 | `leaveCurrentRoom()` + 同名加入踢掉旧身份 |
| 房间不消失/玩家被误删 | 玩家 ID 按 `players.size+1` 生成，退出后 ID 复用撞车，旧连接断开误删新玩家 | 玩家/Bot ID 改单调递增计数器；最后一人离开即销毁房间 |

### 8.3 体验优化

- **炸弹**：初始可连放 3 颗；引信 3s → 2s
- **移动**：短按走一格、按住 300ms 后连走（原来短按也会走两格）；格子间移动改 90ms 补间动画
- **踢炸弹**：方向不再硬编码向右，按当前面向踢；无手套/面前无炸弹时战斗日志有提示
- **地图**：固定石头由永恒偶数网格改为每局随机生成（180° 旋转对称保证公平）
- **画面**：地面/石头/砖块/炸弹/火焰纹理全部重绘加细节
- **分享**：大厅新增"复制玩家邀请链接"+"复制 Agent 接入说明"两个按钮（Agent 说明含完整 WebSocket 协议）

### 8.4 AI 生成美术资源

- **5 个道具图标**（炸弹/火焰/跑鞋/手套/幽灵），透明底 PNG 压缩至 96px
- **4 个角色全套精灵图**：白团子/蓝企鹅/绿恐龙/黄小鸡，每个含 4×4 行走（四方向）、2×2 放炸弹、2×2 被炸，共 12 张
- 原图经 PIL 自动去水印、去分割线、逐帧居中裁剪；加载失败自动回退程序化纹理
- 角色侧面朝向各不相同（企鹅朝左、恐龙一行朝左一行朝右），代码按角色元数据自动翻转

## 九、第2轮迭代心得

### 9.1 顶层脚本必须有"防爆"意识

本次两个最大的 bug（按钮全死、黑屏）都是**一处未捕获异常让整个脚本/场景停摆**。UI 绑定统一走防御式 `$on()` 后，同类问题从"全站瘫痪"降级为"一条警告"。浏览器端没有进程守护，容错必须内建在代码里。

### 9.2 ID 生成不要用"集合大小+1"

玩家退出后 `players.size+1` 会复用旧 ID，异步的断连事件拿着旧 ID 删了新玩家——这类 bug 只在"退出→重进"的时序下触发，单测很难覆盖。单调递增计数器是零成本正解。

### 9.3 无头浏览器是前端调试利器

黑屏问题靠看代码猜了很久方向，用 agent-browser 注入 `window.onerror` 探针后 30 秒定位到 `sprite.add is not a function`。前端疑难 bug：**先抓真实异常，再谈修复**。

### 9.4 AI 生成游戏素材的关键不在"生成"在"对齐"

AI 出的 sprite sheet 网格间距不一、有分割线和水印，直接用必翻车。PIL 流水线（去水印→内容 bbox→等分网格→逐帧居中→统一尺寸）把不可控的生成结果变成标准资产。另外角色侧脸朝向 AI 不听指令，用"每角色记录朝向元数据 + 代码自动翻转"消化这个不确定性，比反复重生成划算。

### 9.5 外部 CDN 是单机项目的单点故障

一次 Phaser CDN 抖动就让整个游戏不可用。本地项目/局域网游戏应零外部依赖，1.1MB 的本地化成本换来永久确定性。

### 10.2 项目总结与心得

- **"本地优先"**：局域网游戏必须零外部依赖，从 CDN 回退到本地 Phaser，再到固定端口、删除测试脚本，本质都是消除环境不确定性。
- **"服务端权威"的代价**：踢炸弹方向、玩家皮肤数据等客户端本地状态容易与服务端不一致，任何需要双方同步的行为都应走服务端校验。
- **素材即代码**：无美术阶段下，程序化纹理是MVP的核心手段，但"同一套代码画四种风格"需要刻意差异化，否则玩家感受不到地图变化。
- **防御式编程**：按钮绑定、剪贴板复制、WebSocket连接都要有降级路径，浏览器环境碎片化，不能假设API一定存在。

## 十二、本轮心得总结

### 12.1 服务器能跑是第一优先级

一个 class 级的重复代码块就让整个项目瘫痪。`node --check` 应该在每次编辑后自动执行——2 秒的成本，省去无限猜疑。

### 12.2 Phaser 动画管线有三条暗坑

1. **纹理切换**：`play(anim)` 会改 sprite.texture，`stop()` 不会还原
2. **class 内方法名不是变量**：`update._flag` 在 class method 内是 ReferenceError
3. **精灵图缩小渲染**：`setTint` 会染坏半透明边缘，AI 素材不能用

### 12.3 随机 ≠ 可用

地图石头必须 BFS 验证连通性。纯随机生成看似"足够随机"，但 13×11 网格上 28 个石头足以封死通路。生成后验证 + 失败重试，10 行代码换永久确定性。

### 12.4 前端崩溃必须"单点隔离"

浏览器没有进程守护，一处未捕获异常就能让整个场景停摆。**UI 绑定防爆、动画代码 try-catch、玩家循环 catch**——三层防护把"全站瘫痪"降为"一条警告"。

### 12.5 缓存破坏是调试第一课

修改 JS 后浏览器不加载新版本，误判"修了没用"。`?v=b11` 省去无数次误判和重复劳动。

### 12.6 自动化的价值

`?watch=1` 从"建房间→等人→准备→开始"四步变为零步。自动化不是锦上添花——当用户需要反复测试时，它就是续航。
